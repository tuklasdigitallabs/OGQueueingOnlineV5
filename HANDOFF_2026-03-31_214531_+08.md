# QSys Online Handoff

Updated on: 2026-04-07 18:30 +08

Where we stopped:
- Staff business date rollover was fixed. `ensureBusinessDate()` now advances stale persisted dates to the current Manila date on request, not only through the 30-second watcher.
- Guest ticket page now shows the guest's current waiting position within their specific pax group. It is computed live from `/api/guest/ticket` and updates when queue state changes/refetches.
- Display media upload now uses FFmpeg conversion. Admin uploads common video files, then the server saves a web-safe MP4 (`H.264`, `AAC`, `yuv420p`, even dimensions, `faststart`).
- Docker image now installs `ffmpeg` so VPS media uploads can transcode inside `og-qsys-app`.
- Cloud media playback was fixed for the local Electron display by allowing cross-origin media responses (`Cross-Origin-Resource-Policy: cross-origin`, `Access-Control-Allow-Origin: *`).
- Display debug logs were quieted and the portrait close button now tries the local Electron bridge before the staff-authenticated close endpoint.

Verified during handoff:
- VPS health endpoint returned `currentBusinessDate: 2026-04-07`.
- Re-uploaded cloud media generated `GLO-websafe.mp4`.
- `ffprobe` inside Docker showed the converted media as:
  - Video: `h264`, profile `High`, `yuv420p`, `576x1024`
  - Audio: `aac`, profile `LC`
- Public media response headers showed:
  - `Content-Type: video/mp4`
  - `Cross-Origin-Resource-Policy: cross-origin`
  - `Access-Control-Allow-Origin: *`
- User confirmed the video is showing.

Operational notes:
- Existing previously uploaded videos are not automatically converted retroactively. Delete and re-upload them from Admin > Media so the upload path creates `*-websafe.mp4`.
- Local-file media mode still depends on the local Electron/Chromium decoder unless a separate local-file conversion/import path is added. Current FFmpeg conversion applies to admin cloud/media-library uploads.
- If public `/qsys` shows `502` after deploy but container health is OK, check Nginx reload/network resolution. A prior transient 502 came from old Nginx workers using a stale upstream IP and resolved after reload.

What needs to be worked on next:
1. Decide whether local-file media selection should also auto-convert files before playback.
2. Continue reviewing staff/admin session isolation if any leakage is still reproducible.
3. Re-test ticket waiting position under call-next, override call, skip, clear-called, and undo flows.
